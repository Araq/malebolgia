However, after using it for several projects I noticed I was doing the same things every time I wrapped a C project. So I dug into <code class="highlighter-rouge">c2nim</code> and have made a number of improvements and features to automated things. I’ve also fixed a number of issues and <code class="highlighter-rouge">c2nim</code> can now parse the entirety of various system headers and handles things like function pointers with function pointers.</p> <p>One of my favorite is the <code class="highlighter-rouge">--reorderComments</code> option. It takes the common C pattern of putting docs before a function and converts it to Nim’s comment style. This alone means that 90% of imported docs will become proper Nim docs associated to the right function! Well mostly. ;) Another favorite is <code class="highlighter-rouge">--stdints</code> that converts modern C’s <code class="highlighter-rouge">int32_t</code> to the proper Nim equivalent.</p> <p>Note that all options should now be available from the CLI and the c2nim pragma’s. Also, I’ve added support for using directives in the form of <code class="highlighter-rouge">#pragma c2nim &lt;option&gt;</code>. This let’s you add c2nim annotations directly in normal C code and they’ll be ignored by your compiler.</p> <p>If others have ideas for improvements, create an issue. I’m doing this mainly for my own projects, but I like the idea of making c2nim basically be capable of automatically wrapping all C code. C++ has also gotten much better as well, but remains tricky. I’ve also got another project in the works to automate running <code class="highlighter-rouge">c2nim</code> on larger projects!</p> <p>Here’s an (incomplete) list of new options (some newer, some older):</p> <ul> <li>importDefines import C defines as procs or vars with <code class="highlighter-rouge"><span class="p">{</span><span class="err">.importc.</span><span class="p">}</span></code></li> <li>importFuncDefines import C define funcs as procs with <code class="highlighter-rouge"><span class="p">{</span><span class="err">.importc.</span><span class="p">}</span></code></li> <li>reorderComments reorder C comments to match Nim’s postfix style</li> <li>mangle:PEG=FORMAT extra PEG expression to mangle identifiers, for example <code class="highlighter-rouge">--mangle:'{u?}int{\d+}_t=$1int$2'</code> to convert C <stdint.h> to Nim equivalents</stdint.h></li> <li>stdints Mangle C stdint’s into Nim style int’s</li> <li>delete:IDENT option to delete nodes with matching idents for procs, types, or vars</li> <li>mergeBlocks merge similar adjacent blocks like two let sections</li> <li>mergeDuplicates merge duplicate proc definitions, etc</li> <li>anonymousAsFields turn C anonymous unions into regular Nim fields - still a C union so caveat emptor</li> </ul> <h2 id="nimitheme"><a href="https://github.com/neroist/nimitheme">nimitheme</a></h2> <h4 id="author-jasmine-aka-neroist">Author: <a href="https://github.com/neroist">Jasmine aka Neroist</a></h4> <p><a href="https://github.com/neroist/nimitheme">nimitheme</a> is a nimble library that enables easy styling of <a href="https://github.com/pietroppeter/nimib">nimib</a> documents. For more information consult the manual about <a href="https://nim-lang.github.io/Nim/manual_experimental.html#strict-definitions-and-nimout-parameters-nimout-parameters">experimental features</a>.</p> <h2 id="strict-effects">Strict effects</h2> <p><code class="highlighter-rouge">--experimental:strictEffects</code> are now always enabled. Strict effects require callback parameters to be annotated with <code class="highlighter-rouge">effectsOf</code>:</p> <div class="language-nim highlighter-rouge"><pre class="highlight"><code> <span class="k">func</span> <span class="n">sort</span><span class="o">*[</span><span class="n">T</span><span class="o">]</span><span class="p">(</span><span class="n">a</span><span class="p">:</span> <span class="k">var</span> <span class="n">openArray</span><span class="o">[</span><span class="n">T</span><span class="o">]</span><span class="p">,</span> <span class="n">cmp</span><span class="p">:</span> <span class="k">proc</span> <span class="p">(</span><span class="n">x</span><span class="p">,</span> <span class="n">y</span><span class="p">:</span> <span class="n">T</span><span class="p">):</span> <span class="kt">int</span> <span class="p">{.</span><span class="n">closure</span><span class="p">.},</span> <span class="n">order</span> <span class="o">=</span> <span class="n">SortOrder</span><span class="p">.</span><span class="n">Ascending</span><span class="p">)</span> <span class="p">{.</span><span class="n">effectsOf</span><span class="p">:</span> <span class="n">cmp</span><span class="p">.}</span> </code></pre> </div> <p>The meaning here is that <code class="highlighter-rouge">sort</code> has the effects of <code class="highlighter-rouge">cmp</code>: <code class="highlighter-rouge">sort</code> can raise the exceptions of <code class="highlighter-rouge">cmp</code>.</p> <h2 id="forbid-certain-effects">Forbid certain effects</h2> <p>Nim’s effect tracking can now forbid certain effects while being oblivious to others:</p> <div class="language-nim highlighter-rouge"><pre class="highlight"><code> <span class="k">type</span> <span class="n">Effect1</span> <span class="o">=</span> <span class="k">object</span> <span class="k">proc </span><span class="nf">test</span><span class="p">(</span><span class="n">callback</span><span class="p">:</span> <span class="k">proc</span><span class="p">(</span><span class="n">x</span><span class="p">:</span> <span class="kt">int</span><span class="p">)</span> <span class="p">{.</span><span class="n">forbids</span><span class="p">:</span> <span class="o">[</span><span class="n">Effect1</span><span class="o">]</span><span class="p">.})</span> <span class="o">=</span> <span class="n">callback</span><span class="p">(</span><span class="mi">1</span><span class="p">)</span> </code></pre> </div> <p>In this example <code class="highlighter-rouge">test</code> takes a callback that can have any effect except <code class="highlighter-rouge">Effect1</code>. This mechanism can be used to enforce that a subsystem cannot accidentically call into a different subsystem. For example, in a game engine it could be used to ensure that the renderer logic does not call into a scripting layer.</p> <h2 id="unicode-operators">Unicode operators</h2> <p><code class="highlighter-rouge">--experimental:unicodeOperators</code> are now always enabled: Unicode operators like <code class="highlighter-rouge">⊗</code> or <code class="highlighter-rouge">∘</code> can be used by math libraries. S. Eliot</p> </blockquote> <p>This blog post is the beginning of a cost model for the implementation that is available via “Nim devel” aka Nim version 2.</p> <p>This implementation was designed for <strong>embedded, hard real-time systems</strong>. Generally speaking, assuming you have enough RAM (which is about <strong>64 kB</strong>) all of Nim’s language features are supported – including exception handling and heap-based storage. The implementation of these features also works on bare metal, without an operating system.</p> <h2 id="heap-based-storage">Heap-based storage</h2> <p>Why the focus on embedded, hard real-time systems? Because when you do these well you can also do everything else well! The algorithms used are oblivious to the heap size: Nim’s memory management works well with a 64 kB sized heap but also scales to a 16 gigabyte heap, for example.</p> <p>Memory can be shared effectively between threads without copying in Nim version 2. The complexity of allocation and deallocation is O(1) and Nim’s default allocator is completely lock-free. Fragmentation is low (average is 15%, and worst case is 25%) as it is based on the <a href="http://www.gii.upv.es/tlsf/">TLSF</a> algorithm. The lock-free ideas have been taken from <a href="https://github.com/microsoft/mimalloc">mimalloc</a>.</p> <p>That means that the cost of <code class="highlighter-rouge">new(x)</code> or <code class="highlighter-rouge">ObjectRef(fields)</code> is O(1) for the allocation and O(<code class="highlighter-rouge">sizeof(x[])</code>) for the required initialization. If the object constructor initializes every field <em>explicitly</em> the <em>implicit</em> initialization step is optimized out.</p> <p>The cost of destruction of a subgraph starting at the root <code class="highlighter-rouge">x</code> is O(N) where N is the number of nodes in the subgraph. A subgraph of acyclic data is immediately destroyed when the refcount of its root reaches 0.</p> <p>Cyclic data is harder to reason about and best avoided. Nevertheless, if you end up creating cyclic data the system remains “deterministic” and responsive at all times. You can influence the cycle collector via a new API:</p> <div class="language-nim highlighter-rouge"><pre class="highlight"><code><span class="k">proc </span><span class="nf">GC_runOrc</span><span class="o">*</span><span class="p">()</span> <span class="sd">## Forces a cycle collection pass. The runtime depends on your program</span> <span class="sd">## but it does not trace acyclic objects.</span> <span class="k">proc </span><span class="nf">GC_enableOrc</span><span class="o">*</span><span class="p">()</span> <span class="sd">## Enables the cycle collector subsystem of `--mm:orc`.</span> <span class="k">proc </span><span class="nf">GC_disableOrc</span><span class="o">*</span><span class="p">()</span> <span class="sd">## Disables the cycle collector subsystem of `--mm:orc`.</span> </code></pre> </div> <p>The idea is that you can schedule a cycle collection when it is “convenient” for your program. In other words when your program is currently not busy. However, in my experiments with Nim’s async event loop I saw no benefits in doing so. The lesson to take away here is <strong>“relax, you’ll be fine”</strong>.</p> <h2 id="deterministic-exception-handling">Deterministic exception handling</h2> <h3 id="subtype-checking-in-o1">Subtype checking in O(1)</h3> <p>Starting with version 2 the <code class="highlighter-rouge">of</code> operator, which is also used implicitly in the <code class="highlighter-rouge">except E as ex</code> construct, is finally as fast as it should be: It’s a range check followed by a memory fetch. The cost is O(1).</p> <p>Nim’s exceptions are based on a good old-fashioned type hierarchy that supports run-time polymorphism. The different exception classes can vary in size. As such, exceptions are allocated on the heap. However, it is possible to preallocate them. The standard library does not do this yet – pull requests are welcome!</p> <h3 id="goto-based-exception-handling">Goto-based exception handling</h3> <p>When you compile to C code, exception handling is implemented by setting an internal error flag that is queried after every function call that can raise. <code class="highlighter-rouge">setjmp</code> is not used anymore. To improve the compiler’s abilities to reason about which call “can raise”, make wise use of the <code class="highlighter-rouge">.raises: []</code> annotation. The error path is intertwined with the success path with the resulting instruction cache benefits and drawbacks.</p> <p>The involved costs are about 2-4 machine instructions after a call that can raise. (There are known ways to optimize this further into one instruction for the most common architectures.) This overhead is annoying but at least it is optimized out for tail calls.</p> <h3 id="table-based-exception-handling">Table-based exception handling</h3> <p>When you compile to C++ code, the C++ implementation of exception handling is used – typically it is based on exception handling tables.</p> <h3 id="which-one-is-better">Which one is better</h3> <p>It depends on your program which implementation strategy performs better.</p> <p>There are rumors that a table-based exception implementation lacks “predictable” performance and so should not be used for hard real-time systems. If these rumors are still true then the goto-based exception handling should be preferred.</p> <h2 id="collections-and-their-costs">Collections and their costs</h2> <h3 id="arrays-objects-tuples-and-sets">Arrays, objects, tuples and sets</h3> <p>These are mapped to linear sections of storage, directly embedded into the parent collection. That means that if no <code class="highlighter-rouge">ref</code> or <code class="highlighter-rouge">ptr</code> indirections are involved, they are allocated on the stack – this is nothing new, it was always true for every Nim version and memory management mode.</p> <p>The reason why these can be embedded directly is simple: They are of a fixed size that is known at compile time.</p> <p>Flexible buffer handling can be done with <code class="highlighter-rouge">openArray</code> which is a <code class="highlighter-rouge">(pointer, length)</code> pair. Both arrays and sequences can be passed to a parameter that takes an <code class="highlighter-rouge">openArray</code>.</p> <h3 id="seqs-and-strings">Seqs and strings</h3> <p>Seqs and strings are <code class="highlighter-rouge">(len, p)</code> pairs in which <code class="highlighter-rouge">p</code> points to a block of memory, sometimes called “payload”. The payload contains information about the available capacity followed by the elements, which are stored in order with no further indirections.</p> <p>Nim does not implement C++’s “small string optimization” (SSO) for the following reasons:</p> <ul> <li>Strings and seqs in Nim are binary compatible: <code class="highlighter-rouge">cast</code>‘ing between them is supported.</li> <li>SSO makes <em>moves</em> slightly slower and Nim is good at moving data around rather than copying.</li> <li>SSO makes the performance harder to predict as small strings are significantly faster to create than long strings (for which the storage needs to be requested from an allocator). SSO also implies that the number of memory indirections differs between long and short strings.</li> </ul> <p>Instead, string <em>literals</em> in Nim cause no allocations and can be shallow copied in an O(1) operation.</p> <p>If you disagree with this design choice and want to have strings that do SSO, there are external packages available (<a href="https://github.com/planetis-m/ssostrings">ssostrings</a>, <a href="https://github.com/metagn/shorteststring">shorteststring</a>).</p> <h3 id="hash-tables">Hash tables</h3> <p>Most of Nim’s standard library collections are based on hashing and only offer O(1) behavior on the <em>average</em> case. Usually this is not good enough for a hard real-time setting and so these have to be avoided. In fact, throughout Nim’s history people found cases of pathologically bad performance for these data structures. Instead containers based on BTrees can be used that offer O(log N) operations for lookup/insertion/deletion. A possible implementation can be found <a href="https://github.com/nim-lang/fusion/blob/master/src/fusion/btreetables.nim">here</a>.</p> <h2 id="threads-locks-and-condition-variables">Threads, locks and condition variables</h2> <p>For better or worse, Nim maps threads, locks and condition variables to the corresponding POSIX (or Windows) APIs and mechanisms. This means their costs are not under Nim’s control. Using an operating system designed for hard real-time systems is a good idea.</p> <p>If your domain is not “hard” real-time but “soft” real-time on a conventional OS, you can “pin” a thread to particular core via <code class="highlighter-rouge">system.pinToCpu</code>. This can mitigate the jitter conventional operating systems can introduce.</p> <h2 id="other-gotchas">Other gotchas</h2> <p>When targeting embedded devices there are many platform-specific knobs and quirks that are beyond the scope of this document. You need to be aware that Nim’s default debug mode is probably too costly so right away you should use <code class="highlighter-rouge">-d:release</code> or a combination of switches like <code class="highlighter-rouge">--stackTrace:off --opt:size --overflowChecks:off --panics:on</code> not to mention the selection of your CPU and OS and setting up a C cross compiler.</p> <p>Feel free to join <a href="https://discord.com/channels/371759389889003530/756920870525730947">Nim’s discord embedded channel</a> and ask for help!</p> <h2 id="conclusion">Conclusion</h2> <p>Nim’s new implementation is excellent for embedded devices, but it is the nature of constrained devices to need specialized solutions like custom containers. For example, a specialized growable array container could save memory if it lacks a “capacity” field and uses only a 16 bit integer to track the current length. Custom containers are easy to create in Nim and they work well together with the builtin constructs because they speak a common <a href="https://nim-lang.org/docs/destructors.html">protocol</a>.</p> <p>How to write an array that can grow at run time without storing the capacity is left as an exercise for the reader.
